# Foodify - о чем приложение?

В первую очередь, я хотел сделать для себя приложение, которое помогает мне в повседневной рутине. В тот момент, для меня было немаловажно оценить КБЖУ продукта, какие есть в нем Е-добавки и полезный ли в нем состав. В дальнейшем, к общему смыслу, я добавил поддержку карт лояльностей, список покупок, а также сравнение продуктов по самым важный метрикам. Отдельная радость была в том, что Роскачество (у которого, если они присутствуют, я беру данные по качеству продукта) есть открытое и качественное API (https://rskrf.ru/about/dev/), благодаря которому в дальнейшем планирую расширить функционал приложения. 

Самая сложная часть - поиск продукта по штрих-коду. К сожалению, в России нет единой базы штрих-кодов товаров со всеми необходимыми значениями. Поэтому я старался брать несколько относительно достоверных источников. Примерная архитектура такая: <img width="500" height="700" alt="примерная архитектура поиска по ШК" src="https://github.com/user-attachments/assets/fda56544-5505-4b78-98f4-33c86fa6a35e" />

Старался делать все через паттерн Chain of Responsibility. Если по пунктам, то сначала был IProductDataProvider:
1. Через Zxing.MAUI беру ШК-товара (пока EAN-13, но потом и DataMatrix Честного знака добавлю)
2. Отправляю ШК черезз официальное API в Роскачество. У Роскачества есть довольное детальные обзоры на товары, с ссылками на исследования.
3. Если данных в Роскачестве нет, то перехожу к Национальному каталогу (как я понял, этот сервис и должен быть агрегатором всех продуктовых товаров, потому что если сканировать ШК в официальном приложении Честного знака - ссылка веден именно на этот сайт. Но нюанс в том, что половина всех товаров, судя по всему, скрыта). Если данные есть, то тут тоже хорошие данные по продуктам и есть детальные характеристики особенно по молочке и воде.
4. Если данных в Нац.каталоге нет, то перехожу в OpenFoodFacts. У них есть удобное официальное API (https://openfoodfacts.github.io/openfoodfacts-server/api/), но вся информация сайта пополняется пользователями, поэтому, к сожалению, в России не так хорошо заполнены карточки товаров и в целом часто отсутствуют.
5. Последний шаг - это через базы ШК узнать наименование товара и попробовать через POST-запрос в Яндекс.Еду получить карточку товара с описанием. Этот шаг самый спорный, потому что из открытых баз данных названия бывают некорректные(barcode-list и disai часто могут дают неверные названия товаров, которые неверны по своей сути, либо не воспринммаются поисковиком Яндекс.Еды), а также в целом отсутствовать. 

По-итогу, если данные находились, я через ICompositionExplanationProvider, смотрел описание и состав продукта и искал добавки. Сама база данных (json файл) Е-добавок сформировал через парсинг сайта proe.info, у них есть довольно детальное описание каждой добавки, а также синонимы к ней, поэтому точность поиска добавок в продукте очень велика и практически всегда он все добавки находит и выдает по ним информацию. 

Отдельно стоит отметить, что провайдер с отзывами пока не сделан. Во многом это потому, что у меня пока нет четкой картины видения это системы. В целом, идея в том, чтобы брать отзывы из сайтов Перекрестка, Ленты, Метро, Мегамаркет и т.д, т.е, у крупных магазинов с огромной покупательской базой. Если поотлавливать fetch/XHR у этих сайтов и потом поэкспериментировать и пробрасывать через Postman, то у многих несложно получить общую оценку и в целом отзывы. Основная проблема - это тема из п.5, нет единого наименования товара у открытых баз данных + каждый магазин дает свое относительно уникальное название (добавляет дополнительное описание товара к наименование или тип упаковки). Самые простые по поиску оказались Metro, Spar, Лента и Магнит. Пятерочка и Перекресток требует куки с сессиией, которую часто нужно менять. Мегамаркет вообще за 2-3 запроса в минуту уже блокирует, как и Вкусвилл. На крайний случай - парсинг IRecommend и Otzovik.
